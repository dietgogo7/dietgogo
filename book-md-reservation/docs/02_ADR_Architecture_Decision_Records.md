# 기술적 의사결정 기록 (ADR - Architecture Decision Records)

이 문서는 시스템의 주요 아키텍처 및 라이브러리 선정 과정에서 발생한 논의 사항과 확정 내용을 기록합니다.

---

## ADR 1. 프론트엔드 환경 제약: 순수 JS & jQuery 적용
*   **상태:** 확정 (Accepted)
*   **배경 (Context):** 
    백엔드는 ASP.NET MVC 5와 MSSQL 스택으로 사전에 고정되어 있습니다. 개발 생산성 및 인수인계의 용이성을 고려하여, 복잡한 Webpack/Babel 빌드 과정이 필요한 최신 프레임워크(React 등) 대신 즉시 IIS 서버에 업로드 후 동작이 가능한 스택을 채택해야 했습니다.
*   **결정 (Decision):** 
    UI 컴포넌트 렌더링 및 통신에는 jQuery 3.x를, 비즈니스 로직 작성에는 순수 자바스크립트(ES5)를 채택합니다.
*   **결과 파급효과 (Consequences):** 
    SPA 프레임워크가 제공하는 강력한 상태 관리(State Management) 리액티비티(Reactivity)가 없어 DOM 조작 코드가 늘어날 수 있습니다. 이를 보완하기 위해 역할 단위로 JS 파일(`calendar.js`, `holidays.js`, `app.js`)을 분리하여 관심사를 나눴습니다.

## ADR 2. UI 트랜지션 구조: 단일 페이지 다중 모달/레이어 트랜지션
*   **상태:** 확정 (Accepted)
*   **배경 (Context):** 
    사용자가 '예약 신청'이라는 하나의 완결성 있는 목적을 달성하기 위해 여러 단계를 거치게 됩니다. 페이지 간 하드 리로드(이동)가 발생하면 UX가 심하게 단절됩니다.
*   **결정 (Decision):** 
    `index.html` 단일 페이지 앱(SPA) 스타일을 취합니다. 크게 MD 우선 선택 플로우와 날짜 우선 선택 플로우를 2개의 컨테이너에 숨겨 두고, 버튼 클릭 시 자바스크립트를 이용해 특정 div만 `display: block`으로 바꾸고 애니메이션을 줍니다.
*   **결과 파급효과 (Consequences):** 
    라우팅 관리가 필요 없어 프론트엔드가 몹시 가벼워집니다. 반면, 모든 예약 단계의 DOM 트리(보이지 않는 엘리먼트들)가 초기 로딩 시 한 번에 브라우저에 마운트되어 있어야 하는 단점이 있습니다.

## ADR 3. 스케줄 동적 관리 방법: 서버 종속성 완화를 위한 Client Config 분리
*   **상태:** 확정 (Accepted)
*   **배경 (Context):** 
    운영 도중 예약 시간대(14시 -> 15시)가 변하거나 미팅 시간 단위(15분 단위 -> 30분 단위)가 빈번히 바뀔 수 있으며, 명절 등 비정기적인 휴무 처리가 매년 발생합니다.
*   **결정 (Decision):** 
    이러한 메타 정보를 하드코딩하지 않고, 자바스크립트 전역 객체인 `SCHEDULE_CONFIG` 와 `holidays.js` 라는 별도 데이터 테이블로 분리했습니다. 
*   **결과 파급효과 (Consequences):** 
    추후 어드민 페이지(ASP.NET)에서 관리자가 '운영 시간'이나 '휴일' 테이블을 수정하면, 해당 데이터를 JSON 형태로 해당 자바스크립트 변수에 내려주기만 하면 클라이언트 사이드 검증 로직이 수정 없이 바로 반영되는 유연함을 획득했습니다.
